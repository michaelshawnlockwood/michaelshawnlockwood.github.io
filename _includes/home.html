About

I have been a database developer my entire adult life. I began with Microsoft Access while in college. Naturally, I migrated to SQL Server using Microsoft's Data Transformation Services (DTS), earning my MCDBA certification in 2006. SQL Server and SSIS have been my foundation for more than 2 decades.

Once Walmart Health's acquisition of MeMD was final, I took the opportunity to invest in furthering my education and improving upon my technical skills through independent project work. One outcome is a real-world multi-billion row public data engineering pipeline focused on large-scale data ingestion, validation, and analytics across SQL Server and PostgreSQL, separately for a number of reasons. I use Python and PowerShell for converting parquet files into PSV format, then bulk import into relational tables. GitHub and VS Code provide version control and development workflow. Airflow is used to orchestrate repeatable pipelines, while Databricks supports distributed analytics. Results are published for visualization in Power BI, ensuring reproducibility and automation end-to-end.

I enjoy working in fast-paced environments and have many years of experience in healthcare and other transaction-heavy industries. While I am a firm believer in mastering core platforms and technologies rather than a generalist of many, I’ve also taken time to learn and experiment with a variety of others—Python, PowerShell, JavaScript, D3.js, Power BI, ASP.NET Core, Identity Server, Node.js, Angular, CSS/Sass, and even MongoDB. These explorations help me stay adaptable and give me perspective, many of which I continue to use in my recurring development and deployment cycles.

What I’m Looking For

I’m seeking not only the right role but also the right environment—one where leadership provides clear guardrails without suffocating innovation, where policies are practical rather than bureaucratic, and where ideas from any level can make their way to the table. In that kind of culture, I contribute best: combining 20+ years of SQL Server expertise with a bias for performance, automation, and verifiable, reliable results.

I’m targeting Senior DBA / Database Engineer roles that require deep SQL Server strength—query tuning, indexing strategy, performance and wait analysis, and HA/DR (Always On availability groups, FCIs, replication, log shipping)—with standardized change control and real observability (tested backups, rehearsed restores, explicit RPO/RTO). SSIS is a core tool; I’ve built enterprise-class ETL pipelines with it. I support DACPAC-based deployments via Visual Studio (SSDT). T-SQL is my second language. I design application-ready result sets—procedures, functions, views, window functions—and, when appropriate, **automate and instrument** the platform.

I’m evolving with the industry—engaging Azure SQL and SQL Server on Azure VMs, AWS, Airflow, and Databricks—and I’ve added PostgreSQL to my toolkit. My working sandbox is practical and current: VS Code; GitHub; Docker for disposable labs; VirtualBox Linux VMs (Debian/Ubuntu) for PostgreSQL and tooling; PowerShell and Bash for scripting and automation; plus selective .NET/Identity Server, and REST APIs (Postman). I’ll use front-end tooling (Angular/React/Sass/SVG) when it helps instrument, diagnose, or deliver—and to present, test, and, most importantly, understand. The through-line is straightforward: ship fast, safe, measurable changes to production databases. Ultimately, the broader toolchain is how I deliver; my core work lives in the data tier.

By “instrument,” I mean:

  • Procedure-level telemetry: log start/end, duration, row counts, error codes, and a correlation ID (via SESSION_CONTEXT) into a lean telemetry table.
  • Extended Events & Query Store: capture deadlocks, timeouts, and plan regressions; baseline known-good plans; surface p95 durations.
  • Wait/IO trending: scheduled DMV snapshots (wait stats, query stats, file/IO stalls) to trend and compare pre/post change.
  • Backup/DR probes: record backup/restore durations and last verified restore; alert on RPO/RTO breaches.
  • ETL/job observability: per-step durations, retries, and data-quality checks (rowcount reconciliations, hash totals) for SSIS and SQL Server Agent jobs.
  • Alerts & dashboards: route severity errors, long-running queries, and blocking chains into monitoring for paging/visuals.

Impact Highlights

90‑minute Outage Rescues

At Grand Canyon University, I resolved a severe performance incident in under 90 minutes that had originally been allotted two weeks for triage. I quickly isolated the bottleneck, applied corrective measures, and stabilized the system. With the crisis resolved, I used the remaining time to design and implement a data-growth forecasting utility that gave the university greater visibility into long-term capacity planning.

During a production outage at FlipSwitch, application performance collapsed after a key index was accidentally dropped. Because I had proactive monitoring in place—tracking disk queue length—I was alerted immediately before the full collapse began. I quickly diagnosed the missing index, scripted its recreation, and restored stability in under 90 minutes. The incident reinforced the value of proactive monitoring of performance metrics and resources, which enable rapid response and safeguard business continuity.

Director who still ships

Led at OnePoint Patient Care for 5+ years while staying hands-on: built and maintained HL7 integrations end-to-end, tuned high-volume OLTP workloads, and delivered reliable releases under real-world pressure. I owned the data tier while running a small, high-performing team—prioritizing a tight backlog, code reviews, and on-call readiness. When incidents surfaced, I led the response and then hardened the system (monitoring, automated tests, and runbooks) to prevent recurrence. Leadership meant clearing roadblocks and setting standards, not stepping away from the keyboard.

Automation mindset

I build tools that take noisy, fragile routines and make them quiet and predictable. My utilities handle adaptive index maintenance, statistics refresh, CHECKDB scheduling with alerts, deadlock/blocked-session capture, job-health monitoring, and restore verification—idempotent by design with clear logs and guardrails. My current project, SQLAgent007, extends this approach across SQL Server, Azure SQL, and PostgreSQL with pluggable collectors, JSON-based configs, dry-run/safe modes, and opinionated defaults for production safety.

Production pragmatist

Bias for observability, repeatability, and low-drama ops: baseline before changes, deploy with rollback plans, and prefer small, reversible steps over risky “big bang” moves. I instrument with Query Store/DMVs/Extended Events, track waits and key counters, and keep runbooks current so on-call is calm, not chaotic. I’ll say “no” to shortcuts that jeopardize uptime—think cavalier NOLOCK usage, unbounded long-running scripts, or schema changes without a cutover plan—and I pair every DR claim with routine test restores and periodic failover drills.
